# predict-by-emh: MVP Providers

> 目标：从已经调研过的全球公开数据源里，缩出**第一版最先接入的 5 个 provider**。
>
> 这里追求的不是“覆盖所有市场”，而是：
> 1. **零 key 或近乎零摩擦**
> 2. **本地电脑直接能跑**
> 3. **能覆盖我们已经验证过的核心问题类型**
> 4. **工程上尽量少，但信息密度尽量高**

---

## 一、选择标准

一个 provider 能进入 MVP，至少要满足以下几点中的大多数：

- **本地可直接调用**：不用复杂认证、不开终端、不依赖重型商业软件
- **信号密度高**：一个源能支持多个问题，不只是服务单一场景
- **代表一种关键数据形态**：事件概率 / 宏观利率 / 衍生品 / 广义价格历史 / 中国市场
- **能和其他源形成互补**：不是纯重复
- **后面可扩展**：未来加 key 或加更专业源时，不会推翻当前设计

---

## 二、MVP Top 5

## 1. `PolymarketProvider`

### 为什么进 MVP

- 这是 `predict-by-emh` 最贴近“直接概率”的数据源
- 对很多问题，它不是结论本身，但几乎总是最有价值的**校准层**
- 本地已验证、零 key、结构化 JSON，接入成本极低

### 最适合回答的问题

- 台海战争概率
- AI 泡沫会不会破
- BTC 价格目标
- 某场战争/选举/政策事件在某个时间窗口内发生的概率

### 它提供的价值

- 事件市场列表
- 盘口价格
- 历史价格
- OI / trade / book

### 不足

- 偏事件驱动，不是宏观骨架
- 散户占比较高，不能单独做结论

---

## 2. `USTreasuryProvider`

> 可以内部同时封装：
> - `Treasury FiscalData`
> - `U.S. Treasury Yield Curve`

### 为什么进 MVP

- 它提供的是整个系统最重要的“**宏观骨架**”
- 黄金、衰退、风险偏好、美元、全球比较类问题，都离不开它
- 零 key、官方、稳定、全球用户都能直接用

### 最适合回答的问题

- 黄金还能不能买
- 美国会不会衰退
- 中国是否在日本化（拿中美/中日利率做比较）
- 风险资产 vs 避险资产的宏观切换

### 它提供的价值

- 10Y / 30Y / 短端利率
- 实际收益率
- 期限结构
- 官方财政/汇率数据

### 不足

- 它是骨架，不提供股票/商品本身价格
- 数据格式偏老，需要清洗

---

## 3. `StooqProvider`

### 为什么进 MVP

- 我们需要一个“**广义价格历史底座**”
- 它虽然不是最优雅的 API，但覆盖足够广，而且零 key
- 很适合本地研究、回测、历史价格验证

### 最适合回答的问题

- 黄金、AI 股票、主题 ETF 的历史走势
- Magnificent 7 vs 等权标普
- 台积电折价、金矿股 vs 金价、行业相对强弱
- 很多“相对表现 / 分化”类信号

### 它提供的价值

- 全球股票
- ETF
- 指数
- 债券
- 外汇
- 一部分宏观/商品价格历史

### 不足

- 更像 CSV 数据源，不像现代 REST API
- 实时性和稳定性不如专业行情服务

---

## 4. `DeribitProvider`

### 为什么进 MVP

- 这是第一版里最值得接的“**专业衍生品层**”
- 它几乎是零成本拿到高价值期权/期货信号的最佳源
- 对 BTC 这种问题，提升不是 10%，而是一个层级的提升

### 最适合回答的问题

- BTC 值不值得买
- crypto 风险偏好是恐慌还是反弹
- 期权 skew / put-call / IV 在如何定价尾部风险

### 它提供的价值

- 期权链
- IV
- open interest
- orderbook
- futures 期限结构
- 实时 websocket

### 不足

- crypto 专用，不能覆盖传统市场
- 需要自己把衍生品字段整理成统一结构

---

## 5. `AkShareProvider`

### 为什么进 MVP

- 如果第一版完全不支持中国市场，很多我们已经验证过的案例就会失去落点
- `AkShare` 虽然底层很多是抓取，但胜在零 key、覆盖广、社区成熟
- 对中国用户尤其重要，但也不妨碍做全球版

### 最适合回答的问题

- 沪深300 是否值得定投
- 中国是否在日本化
- 中国房价代理信号
- 北向资金、融资余额、指数结构

### 它提供的价值

- A股与指数
- 北向资金
- 融资融券
- 期货
- 债券
- 宏观

### 不足

- 稳定性不如官方 API
- 底层依赖多个公共网站，部分接口有抓取脆弱性

---

## 三、为什么不是别的源

### 为什么 `Kalshi` 没进前 5

不是因为它不重要，而是因为它和 `Polymarket` 有一定重叠。

它非常适合作为 **MVP 第 6 个 provider**：

- 它比 Polymarket 更偏美国严肃宏观
- 但如果只取一个“事件概率层”，`Polymarket` 覆盖面更广，演示效果更强

### 为什么 `CoinGecko` 没进前 5

它很有用，但在第一版里，`Deribit` 的信号密度更高。

- `CoinGecko` 更像 crypto 现货底座
- `Deribit` 更像 crypto 专业定价层

第一版要突出 `predict-by-emh` 的特色，应该优先接“**更有预测力**”的那个。

`CoinGecko` 非常适合作为 **MVP 第 7 个 provider**。

### 为什么 `SEC EDGAR` 没进前 5

它很强，但更适合作为第二阶段增强：

- 适合做“超预期但下跌”
- 适合做“高管减持”
- 适合做财报与披露驱动的分析

但它不如前 5 个那样“覆盖一个大信号层”。

### 为什么 `World Bank / Eurostat / ECB` 没进前 5

这些源对全球宏观比较非常好，但第一版优先级不如：

- 事件概率层
- 宏观利率骨架
- 价格历史层
- 衍生品层
- 中国市场层

它们更适合作为 **第二阶段的全球比较增强层**。

---

## 四、MVP 第 6-10 名

如果 Top 5 跑通，下一批建议接入的是：

6. `KalshiProvider`
7. `CoinGeckoProvider`
8. `SecEdgarProvider`
9. `WorldBankProvider`
10. `EurostatProvider`

### 这批的定位

- `Kalshi`：补强事件概率层，提升美国宏观场景质量
- `CoinGecko`：补强 crypto 现货与稳定币基础层
- `SEC EDGAR`：补强个股/公司层信号
- `World Bank / Eurostat`：补强跨国宏观比较

---

## 五、这 5 个 provider 已经能回答什么

| 问题类型 | 核心 provider |
|----------|----------------|
| 战争 / 选举 / 事件概率 | `Polymarket` |
| 黄金还能不能买 | `USTreasury` + `Stooq` |
| BTC 值不值得买 | `Deribit` + （后续可加 `CoinGecko`） |
| AI 泡沫会不会破 | `Stooq` + `Polymarket` |
| AI 还能高速迭代多久 | `Stooq` + `Polymarket` + `USTreasury` |
| 沪深300 值不值得定投 | `AkShare` |
| 中国是否日本化 | `AkShare` + `USTreasury` |
| 台海战争概率 | `Polymarket` + `Stooq` |

这意味着：

> **只靠这 5 个 provider，就已经能覆盖我们目前讨论过的大部分代表性问题。**

---

## 六、推荐的实现顺序

不要 5 个一起写，按信号层顺序来：

1. `PolymarketProvider`
2. `USTreasuryProvider`
3. `StooqProvider`
4. `DeribitProvider`
5. `AkShareProvider`

### 为什么这样排

- `Polymarket`：最快出结果，最容易 demo
- `USTreasury`：先把宏观骨架立起来
- `Stooq`：补上广义价格历史
- `Deribit`：补上最有信息量的衍生品层
- `AkShare`：最后补中国市场，避免第一天就掉进抓取细节

---

## 七、最终结论

如果现在就要开始做 MVP，而不是继续调研，那么**最先接入的 5 个 provider** 我建议就是：

1. `PolymarketProvider`
2. `USTreasuryProvider`
3. `StooqProvider`
4. `DeribitProvider`
5. `AkShareProvider`

它们分别代表了 5 种关键能力：

- **事件概率**
- **宏观骨架**
- **广义价格历史**
- **专业衍生品**
- **中国市场落地**

这 5 个一起，已经足以做出一个有“predict-by-emh 味道”的第一版系统。
